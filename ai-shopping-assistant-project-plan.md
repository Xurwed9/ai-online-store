## 1. Резюме проекта

AI-ассистент для онлайн-магазина, взаимодействующий с пользователем на естественном языке, подбирающий товары через MCP-инструменты и оформляющий заказы. Ниже — план разработки от анализа требований до сдачи проекта, а также точечные улучшения исходного ТЗ: закрыты пробелы в модели данных, наборе инструментов, обработке ошибок и границах ответственности AI.

**Оценка длительности:** 4–6 недель при части времени команды из 2–3 человек (учебный/пет-проект темпа). Ниже разбивка по неделям — подстройте под свой график.

---

## 2. Цели и критерии успеха

| Цель | Метрика |
|---|---|
| Понимание запроса на естественном языке | ≥90% корректного извлечения category/color/size/price на тестовом наборе из 30 фраз |
| Релевантность подбора | Пользователь получает ≤3 релевантных варианта за один запрос без уточнений (если параметры полны) |
| Корректность оформления заказа | 0 заказов без явного подтверждения оплаты |
| Устойчивость к ошибкам | Все сценарии из раздела "Обработка ошибок" покрыты тестами |

## 3. Ключевые пробелы исходного ТЗ и их устранение

### 3.1 Схема базы данных — недостающие элементы

Исходная схема не выдержит даже базовый сценарий. Дополнения:

```
products
  + stock_quantity (int)          -- без него check_stock невозможен
  + sku (string, unique)
  + images (json/array)
  + is_active (bool)               -- для снятых с продажи товаров

users
  + phone (string)
  + created_at

cart
  + created_at / updated_at        -- для очистки "забытых" корзин

orders
  + shipping_address_id (fk)
  + updated_at

payments
  + created_at
  + transaction_id (string)        -- для сверки с платёжным провайдером

-- НОВЫЕ ТАБЛИЦЫ
addresses(id, user_id, line1, city, postal_code, country, is_default)
purchase_history(id, user_id, order_id, product_id, purchased_at)
discounts(id, code, type, value, valid_from, valid_to, min_order_total)
reviews(id, product_id, user_id, rating, comment, created_at)   -- опционально, для рекомендаций
```

Без `stock_quantity` шаг 4 сценария ("AI проверяет наличие на складе") физически нечем выполнять — это критичный пробел, а не деталь на потом.

### 3.2 MCP Tools — недостающий набор

Исходный список (`search_products`, `get_product_details`, `create_order`) не покрывает даже основной сценарий из раздела 2 (там упоминается `check_stock` и `recommend_products`, которых нет в разделе 5 — рассинхрон в самом ТЗ). Полный рекомендуемый набор:

```
# Поиск и подбор
search_products(category, color, size, min_price, max_price, sort_by)
get_product_details(product_id)
check_stock(product_id, size?)
recommend_products(user_id?, based_on_product_id?, limit=3)

# Корзина
add_to_cart(user_id, product_id, quantity, size)
update_cart_item(cart_item_id, quantity)
remove_from_cart(cart_item_id)
get_cart(user_id)

# Заказы и оплата
create_order(user_id, cart_id, shipping_address_id)
confirm_payment(order_id, payment_method)      -- ОТДЕЛЬНЫЙ шаг от create_order
get_order_status(order_id)
cancel_order(order_id)

# Пользователь
get_user_preferences(user_id)
update_user_preferences(user_id, preferences)
```

**Важно:** `create_order` и `confirm_payment` разведены намеренно — это прямое следствие правила "AI не должен подтверждать оплату без явного согласия", которое иначе просто нечем технически обеспечить.

### 3.3 Границы поведения AI — конкретизация

Исходные 3 правила верны, но недостаточно операциональны. Уточнения:

- **Подтверждение оплаты:** AI формирует заказ в статусе `pending`, показывает пользователю сумму и состав, и вызывает `confirm_payment` только после явного текстового подтверждения ("да", "оплатить", "подтверждаю").
- **Альтернативы при отсутствии:** если `check_stock` возвращает 0 — вызвать `search_products` с ослабленными фильтрами (например, убрать цвет или расширить ценовой диапазон на 15–20%) и явно сообщить, что это альтернатива, а не точное совпадение.
- **Неоднозначные запросы:** уточняющий вопрос — один за раз, не более 2 уточнений подряд, иначе предложить топ-3 варианта "на угад" с пометкой "уточните, если не подходит".
- **Границы полномочий:** AI не должен изменять цены, применять произвольные скидки или редактировать данные товара — это вне зоны MCP-инструментов, доступных ассистенту.

### 3.4 Обработка ошибок — сценарии, которых не было

| Сценарий | Ожидаемое поведение |
|---|---|
| Оплата не прошла | Заказ остаётся в `pending`/`payment_failed`, AI предлагает повторить или сменить способ оплаты, заказ НЕ считается оформленным |
| Товар закончился между поиском и заказом | `create_order` возвращает ошибку конфликта → AI повторно вызывает `check_stock`, сообщает пользователю и предлагает альтернативу |
| Товар недоступен в нужном размере при выборе | То же, что 3.3 — расширение поиска с пометкой |
| Двойной клик / повторный запрос на оформление | Идемпотентность на уровне `create_order` (idempotency key на сессию) |
| Тайм-аут платёжного провайдера | Статус `payment_pending`, не блокировать пользователя, дать возможность проверить статус позже (`get_order_status`) |

## 4. Архитектура (высокоуровнево)

```
Пользователь ⇄ Chat UI
                 │
                 ▼
         AI-ассистент (LLM + оркестрация)
                 │  вызывает MCP tools
                 ▼
        MCP Server (бизнес-логика)
                 │
                 ▼
        База данных (products, users, orders...)
                 │
                 ▼
        Внешний платёжный провайдер (мок или sandbox)
```

Рекомендация: платёжный шаг реализовать через мок/sandbox-провайдера (например, Stripe test mode), не через реальные платежи — иначе появляется отдельный пласт compliance-работы, не входящий в скоуп учебного проекта.

## 5. Этапы разработки

### Неделя 1 — Проектирование
- Финализировать схему БД (с учётом п.3.1)
- Спроектировать полный набор MCP-инструментов (п.3.2) и их контракты (входы/выходы/ошибки)
- Написать 15–20 тестовых пользовательских фраз для проверки извлечения параметров
- **Результат:** ER-диаграмма, спецификация API/tools, тестовый набор фраз

### Неделя 2 — Backend и данные
- Развернуть БД, миграции, seed-данные (минимум 30–50 товаров в разных категориях)
- Реализовать MCP-сервер: search/get_details/check_stock
- Реализовать корзину: add/update/remove/get_cart
- **Результат:** работающий MCP-сервер без AI-слоя, покрытый юнит-тестами

### Неделя 3 — AI-слой и оркестрация
- Интеграция LLM с MCP tools (system prompt, tool-calling)
- Реализовать извлечение параметров из свободного текста
- Логика уточняющих вопросов (п.3.3)
- Логика recommend_products (для старта — простой фильтр + сортировка по релевантности)
- **Результат:** ассистент проходит основной сценарий (шаги 1–6 из раздела 2 исходного ТЗ) в чате

### Неделя 4 — Заказы, оплата, ошибки
- create_order → confirm_payment раздельно
- Обработка сценариев из п.3.4
- get_order_status, cancel_order
- **Результат:** полный цикл "запрос → заказ → оплата → статус" работает end-to-end

### Неделя 5 — Персонализация и полировка (опционально, из раздела 7 ТЗ)
- purchase_history + простые рекомендации ("похожие товары")
- Профиль предпочтений пользователя
- UI-полировка чата

### Неделя 6 — Тестирование и сдача
- Регрессионное тестирование сценариев ошибок
- Нагрузочный smoke-тест (10–20 параллельных диалогов)
- Документация: README, схема БД, примеры диалогов
- Демо-запись/презентация

## 7. Технологический стек (рекомендация, не догма)

- **MCP-сервер:** Node.js/TypeScript или Python (FastAPI) — оба хорошо документированы для MCP
- **БД:** PostgreSQL (нужны транзакции для заказов/оплаты) или SQLite для локальной разработки
- **LLM-слой:** Claude через API с tool use
- **Чат-интерфейс:** минимальный веб-чат (React) или CLI для первых итераций — не тратить неделю 1–2 на UI

## 8. Риски

| Риск | Вероятность | Митигация |
|---|---|---|
| Рассинхрон между заявленными и реализованными tools | Высокая (уже есть в исходном ТЗ) | Единый файл-контракт tools, синхронизируемый на неделе 1 |
| LLM "додумывает" размер/наличие вместо вызова check_stock | Средняя | Строгий system prompt: любой ответ о наличии — только после вызова tool |
| Заказ оформляется без подтверждения оплаты | Средняя, но критична | Разделение create_order/confirm_payment на уровне архитектуры (не только промпта) |
| Скоуп расползается на раздел 7 раньше времени | Высокая для учебного проекта | Разделы 7 — только после недели 4, и то опционально |
